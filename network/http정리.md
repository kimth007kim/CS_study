- HTTP 헤더 
클라이언트와 서버가 요청또는 응답으로 부가적인 정보를 전송할수 있게해줌

1 .General header : 요청 및 응답 메시지 모두에서 사용되지만 컨텐츠 자체에는 적용되지 않는 헤더 ex) Date: 현재시간 
 
2. Request header : 페치될 리소스나 클라이언트 자체에 대한 자세한 정보를 포함하는 헤더. 내가 보내는 메세지의 헤더  ex) Host : 서버의 도메인명과 서버의 포트를 지정합니다.

3. Response header : 위치 또는 서버 자체에 대한 정보(이름, 버전 등)와 같이 응답에 대한 부가적인 정보를 갖는 헤더. == 내가 받은 메세지의 헤더 ex) Server: 서버의 소프트웨어 정보, Set-Cookie: 서버측에서 클라이언트에게 세션 쿠키 정보를 설정

4. Entity header: 요청과 응답 모두에 사용가능한, Entity(콘텐츠,body,리소스)에 대한 설명헤더 ex) Content-Type 해당 개체에 포함되는 미디어 타입 정보


- HTTP 상태 코드

1xx informational 리퀘스트를 받아들여 처리중
2xx success 리퀘스트를 정상적으로 처리했음
3xx redirection 리퀘스트를 완료하기 위해서는 추가동작이 필요
4xx client error 서버는 리퀘스트 이해 불가능
5xx server error 서버는 리퀘스트 처리 실패

200 - OK . 요청성공

400 - Bad Request. 요청오류
401 -
403 -
404 - Not Found. 리소스가 없는 상태

500 - Internal Server Error. 서버가 요청 처리를 못함

HTTP의 장점

1. HTTP는 간단하다. 
사람이 읽을수 있도록 간단하게 고안되었기 때문에 그렇다, 테스트하기 쉬워서 진입 장벽이 낮다.

2. HTTP는 확장이 가능하다.
클라이언트와 서버가 새로운 헤더의 합의가 이루어진다면, 언제든지 새로운 기능이 추가될수 있다.

HTTP의 단점

1. 평문이기 때문에 도청이 가능하다.(암호화가 안되어있다.)
TCP/IP를 통해서 HTTP 전송이 이루어진다. TCP/IP는 도청이 가능하기때문에, 암호화가 안되어있는 HTTP의 경우 도청을 쉽게 당한다.
  - 해결책 
    통신 암호화
    콘텐츠 암호화
 
2. 통신상대를 확인하지 않기 때문에 위장이 가능하다.( 누구나 리퀘스트 할수있다)
 HTTP는 누가 리퀘스트를 보내건간에 리스폰스가 반환되는 심플한 구조이다.
 이러한 경우 문제가 될수 있는 상황은

 2-1 위장한 웹서버에 리퀘스트를 보낼 수도있다
 2-2 리스폰스를 반환한곳이 리퀘스트를 보낸 클라이언트인지 아닌지 확인할수 없다.(위장한 클라이언트일수있다)
 2-3 통신하고있는 상대가 접근이 허가된 상대인지 확인할수 없다,
 해결책 
 SSL 로 상대를 확인한다.

3. 완전성을 증명할 수없기떄문에 변조가 가능
 완전성을 증명할수 없다는 것은 리퀘스트나 리스폰스가 발신된후 상대가 수신해도 이것이 변조되었는지 아닌지 확인할 수없기때문이다.
